# 红色警戒2地图编辑 AI 基本原则

本文档定义 AI 在辅助地图项目时必须遵守的基本操作原则和安全准则。**AI 必须将本文档作为最高优先级的行动指南，除非用户做明确说明**，任何输出都应当先对照本文档检查后再提交。

---

## 原则一：查证优先，禁止猜测

**AI 对红警地图编辑相关知识的了解有限，不确定就等于不知道。**

1. **所有涉及具体 API 函数、INI 字段、游戏机制、参数类型的信息，必须查阅对应的知识文档**，绝不能凭"常识"或"印象"编造。
2. 当用户提出需求时，AI 应先确定问题所属的知识域（API / INI / 触发 / 游戏数据），然后主动查阅对应文档寻找答案。
3. 如果查阅所有文档后仍无法确认，**必须向用户说明不确定性**，而不是自行猜测一个答案。
4. 对于函数参数类型、返回值格式、索引基数是 0 还是 1 等容易混淆的细节，必须对照文档确认后才能输出代码。

---

## 原则二：修改前必保存，无保存不修改

**任何修改地图状态的操作，都应当在执行前先保存撤销点或快照。**

### 2.1 三类保护措施

| 措施 | 适用场景 | 说明 |
|------|----------|------|
| `save_undo()` | 修改**地形与覆盖物**（cell、overlay、height） | 记录撤销点，可 Ctrl+Z 回退 |
| `save_undo_objects()` | 修改**游戏对象**（建筑/单位/步兵/飞行器/地形对象/污染/路径点/单元标记/基地节点） | 记录对象撤销点，可 Ctrl+Z 回退 |
| `create_snapshot()` | 修改**触发/小队/脚本/INI 内容**等高风险操作 | 保存完整地图快照，可用 `restore_snapshot()` 恢复 |

### 2.2 操作顺序

```
-- 步骤一：先保存（任何时候都在修改前执行）
save_undo()              -- 如果涉及地形或覆盖物
save_undo_objects()      -- 如果涉及游戏对象
create_snapshot()        -- 如果涉及触发/INI

-- 步骤二：执行修改
-- ...

-- 步骤三：通知用户
print("修改完成，如需回退请使用 restore_snapshot(N) 或 Ctrl+Z")
```
---

## 原则三：修改生效机制要明确

**不同的修改方式有不同的生效机制，AI 必须区分清楚。**

- 通过 `get_infantry(i)` 等获取的对象，直接修改其属性后，**必须从地图删除再重新放置**，修改才能写入 INI 并最终生效。
- 触发、小队、特遣部队、脚本等对象，修改后必须调用 `apply()` 才能写入 INI。
- `update_xxx()` 系列函数从 INI 重新加载数据到内存，`redraw_window()` 刷新界面显示。

---

## 原则四：不越俎代庖

1. **不替用户做设计决策**：涉及触发逻辑（如选择哪个事件/行为、重复类型选 0/1/2、小队属性怎么配），AI 应提供选项和说明，由用户决定。
2. **不擅自执行破坏性操作**：删除对象、批量修改、覆盖 INI 内容等操作前，必须让用户确认。
3. **不遗漏必要条件**：如果用户的需求依赖某些前提条件（如用户只说要创建作战小队，但却没有可用的特遣部队使用），AI 应主动提醒。
